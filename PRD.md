# AI智能餐垫 PRD V1.3

**工程更新版｜补齐 App / 后端技术路线、AI 对话、匿名用户与本地优先策略**

---

## 0. 本次版本更新结论

V1.3 不是重新定义产品，而是把最近几轮工程讨论中已经确定的实现决策正式写进 PRD，让“产品定义、技术路线、AI 接入方式、前后端边界”保持一致。

- 保留 V1.2 的产品边界：实时判断由规则算法完成，AI 不进入实时提醒链路。
- 正式把后端路线调整为 **FastAPI + PostgreSQL + LangGraph**；V1 先用 **FastAPI BackgroundTasks**，不强制上 Redis。
- 正式增加匿名用户模型：App 首次启动生成 `anonymousUserId`，本期不做注册登录。
- 正式增加“**设置本地优先、后端同步**”原则，保障离线和后端异常时仍能正常提醒。
- 正式增加“**提醒文字 / 语音内容可编辑**”要求，并定义为设置模块的必选项。
- 正式增加“**AI 陪伴式对话**”能力位，作为餐后复盘、督促与建议的扩展方向，但不影响首版主闭环。

## 1. 产品定位与边界

### 1.1 一句话定义

AI 智能餐垫是一款通过电子秤重量变化识别用户进食节奏、在吃得过快时进行低打扰提醒，并在餐后输出可信总结、趋势解释与陪伴式建议的智能饮食习惯改善产品。

### 1.2 首版核心价值

- 让用户第一次明确感知“我吃饭真的太快”。
- 让提醒发生在正确时机，且足够轻，不打扰用餐。
- 让单餐与趋势报告足够简单、可信、能执行。
- 让 AI 作为解释层与陪伴层出现，而不是不稳定的判断内核。

### 1.3 非目标

- 首版不做食物识别、热量估算、医疗诊断、营养处方。
- 首版不追求复杂多人共餐场景的高精度。
- 首版不做登录注册和复杂会员体系。
- 首版不依赖云端在线才能完成餐中提醒。

## 2. 当前硬件接入假设（面向 App 开发）

为便于工程推进，本版先把硬件端简化为“一台名为 **BH** 的 BLE 电子秤”。

当前工程假设：

- 设备名：`BH`
- 通信方式：**BLE 广播包**
- 现阶段以**广播包接入**为主，而不是把 GATT 连接链路作为首要依赖
- 已观察到：
  - 6g 时广播数据出现类似 `06`
  - 102g 时广播数据出现类似 `66`
- 当前推断：广播 `manufacturer data` 中已包含重量字段
- 代码实现上不要把协议偏移写死，要保留可调解析能力

---

## 3. MVP 主闭环

首版只要求跑通以下闭环：

**电子秤广播重量 → App 解析重量 → 本地判断开始/结束/过快 → 触发提醒 → 本地记录 → 上传后端 → 生成单餐报告 → 聚合趋势 → AI 给出总结与建议**

补充说明：

- 餐中：依赖本地规则和本地设置，不依赖实时联网。
- 餐后：后端生成规则报告，并异步补充 AI 总结。
- 长期：后端聚合 7 天趋势，AI 可进一步输出趋势解释与建议。
- 扩展：后续在不破坏主闭环的前提下接入陪伴式对话。

## 4. 前端（App）技术路线

### 4.1 技术选型

- Flutter
- BLE 扫描广播方式接入
- `sqflite` 做本地存储
- 本地规则引擎负责实时判断
- 本地缓存负责设置与离线兜底

### 4.2 页面范围：先做 5 个页面

#### 1）设备连接 / 蓝牙调试页
功能：
- 扫描 `BH` 设备
- 显示设备名、MAC、RSSI
- 显示广播原始数据 `rawHex`
- 显示解析后的重量值
- 作为协议验证与调试页面长期保留

#### 2）首页
功能：
- 显示设备在线状态
- 显示最近一餐摘要
- 显示今日已记录餐次
- 提供进入实时页 / 报告页 / 趋势页 / 设置页的入口

#### 3）用餐实时页
功能：
- 实时显示当前重量
- 实时显示当前速度
- 判断是否过快
- 判断是否开始吃饭 / 结束吃饭
- 触发提醒
- 记录提醒次数

#### 4）单餐报告页
功能：
- 显示本餐时长
- 显示总进食克数
- 显示平均速度
- 显示峰值速度
- 显示提醒次数
- 显示一句总结
- 预留 AI 单餐建议展示位

#### 5）趋势 + 设置页
趋势部分：
- 最近 7 天平均速度
- 最近 7 天快吃次数
- 改善率
- 预留 AI 趋势总结展示位

设置部分：
- 提醒开关
- 提醒频率
- 提醒文字内容可编辑
- 语音内容可修改（第一版采用文字 + 系统 TTS）
- 语音试播
- 静音时段
- 本地缓存生效提示

### 4.3 前端核心模块

- **BLE Scan Service**：持续扫描 `BH` 广播包，过滤目标设备并读取 `manufacturer data`
- **Broadcast Parser**：把原始 bytes 转成 `rawHex` 与候选重量字段，解析逻辑可扩展
- **Weight Processor**：滤波、去抖、降采样，形成稳定的重量流
- **Meal Engine**：判断开始吃饭、结束吃饭、当前速度、是否过快、提醒冷却
- **Reminder Service**：读取本地设置后执行提醒，支持文字与系统 TTS 播报
- **Reminder Settings Service**：保存和修改提醒开关、频率、提醒文字、语音开关、静音时段
- **Local DB / Local Cache**：保存匿名用户 ID、设置缓存、餐次记录、重量样本、提醒事件

### 4.4 前端关键实现原则

- App 首次启动生成 `anonymousUserId`，并持久化到本地。
- 设置采用“**本地优先、后端同步**”：启动先读本地，再尝试拉取后端并覆盖缓存。
- 餐中提醒永远依赖本地设置，不依赖网络是否在线。
- 第一版提醒语音采用系统 TTS，用户编辑的是提醒文字内容；系统根据该文字试播和播报。
- 蓝牙调试页属于研发页，但必须保留，因为它是协议验证与排障的第一现场。

## 5. 后端技术路线

### 5.1 总体路线

后端采用：

**FastAPI 主业务后端 + LangGraph AI 工作流 + PostgreSQL 数据存储**

V1 先使用 **FastAPI BackgroundTasks** 处理 AI 报告异步生成，不强制引入 Redis。

### 5.2 业务模块

#### 1）device
负责：
- 绑定匿名用户与设备
- 保存 `deviceName / mac / bindTime`
- 查询当前设备

#### 2）settings
负责：
- 保存和读取用户提醒设置
- 支持本地缓存同步
- 字段至少包括：
  - `reminderEnabled`
  - `reminderFrequency`
  - `reminderText`
  - `voiceEnabled`
  - `quietHours`

#### 3）meal
负责：
- 接收单餐上传数据
- 接收重量样本
- 接收提醒事件
- 接收餐次事件
- 数据归属于 `anonymousUserId`

#### 4）report
负责：
- 返回单餐报告
- 如果 AI 结果未生成，先返回规则报告
- AI 结果生成后可覆盖或补充

#### 5）trend
负责：
- 返回最近 7 天趋势
- 计算平均速度 / 快吃次数 / 改善率

#### 6）chat
负责：
- 陪伴式对话接口
- 为后续“督促 + 建议 + 陪伴”做准备
- 保存 `thread / message / summary / memory` 关联关系

### 5.3 AI 模块（LangGraph）

- **meal_insight_graph**：单餐 AI 总结、单餐建议、低可信度降级
- **trend_insight_graph**：趋势解释、趋势建议、样本不足降级
- **companion_chat_graph**：陪伴式对话、餐后督促、建议生成，读取用户近期餐次与偏好
- **AI 不参与实时链路**：不开餐判定、不结束判定、不实时提醒、不阈值触发

## 6. 用户模型与数据归属

本期不做注册登录，但后端必须识别“数据属于谁”。因此 V1 引入匿名用户模型。

### 6.1 方案

- App 首次启动生成 `anonymousUserId`
- 存在本地
- 后续所有请求都携带该 ID
- 后端按该 ID 归属：
  - device
  - settings
  - meal
  - report
  - trend
  - chat

### 6.2 当前不做的内容

- 手机号登录
- 邮箱登录
- 第三方登录
- 多设备账号同步体系

## 7. 提醒策略与设置设计

### 7.1 首版必选设置项

- `reminderEnabled`
- `reminderFrequency`
- `reminderText`
- `voiceEnabled`
- `quietHours`

### 7.2 设计原则

- 提醒文字内容必须可编辑；这是体验差异化的核心，不是可选项。
- 第一版“语音内容可修改”的实现方式为：用户修改提醒文字，App 使用系统 TTS 试播和播报。
- 提醒服务执行前需判断：提醒是否开启、是否在冷却期、是否处于静音时段。
- 提醒记录必须落本地并可上传后端，以便生成单餐报告与后续优化提醒体验。

---

## 8. 数据流与接口最小口径

### 8.1 数据流

- 广播扫描：App 扫描 `BH` 广播，得到 raw manufacturer data
- 协议解析：App 将 raw data 解析为 `rawHex` 与 `weightGram` 候选值
- 本地判断：Meal Engine 用稳定重量流计算餐次状态与速度
- 本地提醒：根据本地设置执行文字 / 语音提醒
- 数据上传：餐次结束后上传 `meal summary / samples / events`
- 后端生成：规则报告先出，AI 报告异步补充，趋势按 7 天聚合

### 8.2 接口建议（V1）

- `POST /device/bind`
- `GET /device/current`
- `GET /settings`
- `PUT /settings`
- `POST /meal/upload`
- `GET /meal/{mealId}`
- `GET /report/{mealId}`
- `GET /trend/7d`
- `POST /chat/send`
- `GET /chat/thread/{threadId}`

## 9. AI 输出边界

- AI 输入必须是**结构化数据**，不直接读取 BLE 原始时序
- AI 只负责：总结、解释、建议、陪伴式回复
- 低可信度或样本不足时，必须降级为模板文案，不能给强结论
- 输出禁止医疗化措辞，不输出诊断、治疗、处方建议
- 陪伴式对话的身份是“陪伴者 / 督促者”，不是医生，也不是心理咨询师

## 11. 当前版本的最终判断

V1.3 的核心价值，是把“能讲清楚产品”进一步升级为“能直接开工”。

它补齐了：

- 匿名用户
- 本地优先设置
- 提醒文案可编辑
- FastAPI + LangGraph
- AI 陪伴式对话

同时仍然守住首版边界：

**实时链路必须本地稳定，AI 只能在该说话的时候说话。**

### 当前优先验证点

- 产品上：提醒是否有用、报告是否可信、趋势是否让用户愿意继续记录
- 工程上：先把 BLE 广播解析、Meal Engine、本地设置和 meal/report 接口打通
- AI 上：先做单餐与趋势总结，再扩展到陪伴式督促对话
